在写任何一行代码之前，必须先把这件事说清楚——这是整门课的地基。

那 harness 工程师到底在干嘛？先把 harness 拆开看。

它就是这五个东西的加和——工具、知识、观测、行动接口、权限。

工具，是给 agent 一双手：文件读写、Shell、网络、浏览器。设计时要原子化、可组合、描述清晰。

// 最小循环 · 每个 AI Agent 都需要它

THE AGENT PATTERN

User --> messages[] --> LLM --> response | stop_reason == "tool_use"? / \ yes no | | execute tools return text append results loop back -----------------> messages[]

① 模型决定

何时调用工具、何时停止——这是模型的判断，不是代码的硬编码。

② 代码执行

代码只是执行模型的要求：拿到 tool_use block，跑对应 handler，把结果塞回去。

③ 循环不变

后面 19 课都在这个循环之上叠机制——循环本身始终不变。循环属于 agent，机制属于 harness。

好，重头戏来了。这一页是整门课的锚点——THE AGENT PATTERN。

看这个流程图。用户输入变成 messages 数组，发给 LLM，LLM 回一个 response。然后关键的分叉出现了：检查 stop_reason 是不是等于 tool_use。如果是 yes，就执行工具、把结果 append 回去、loop 回去再问一次；如果是 no，就返回文本，结束。

就这么简单。这是最小循环，每个 AI Agent 都需要这个循环。

下面三张卡片讲清楚分工：第一，模型决定——什么时候调工具、什么时候停，是模型在判断，不是代码在硬编码。 第二，代码只是执行——拿到 tool_use block，找到对应的 handler 跑一下，把结果塞回 messages。第三，也是最重要的——后面 19 课都在这个循环之上叠加 harness 机制，但这个 while True 循环本身，从 s01 到 s20 始终不变。

s01 AGENT LOOP 循环 · 模型与真实世界的第一道连接

一个循环 & Bash is all you need

“一个工具 + 一个循环 = 一个 Agent”

痛点

模型能"说出"一条 bash 命令，但它不会自己运行——得有人复制到终端、跑、把输出贴回去。把这个人自动化掉，就是 agent loop。

两个信号驱动一切

  • stop_reason == "tool_use" → 执行工具，喂回结果，继续循环
  • 否则 → 返回文本，结束
# s01 · 最小 agent harness（<30 行） def agent_loop(messages): while True: resp = client.messages.create( model=MODEL, system=SYSTEM, messages=messages, tools=TOOLS) messages.append({"role":"assistant", "content": resp.content}) if resp.stop_reason != "tool_use": return results = [run_bash(b.input["cmd"]) for b in resp.content if b.type == "tool_use"] messages.append({"role":"user", "content": results})

s01，整门课的起点。格言是——一个工具 + 一个循环 = 一个 agent。

先说痛点。你跟 LLM 聊天，它说"你可以跑一下 ls -la 看看"。但模型自己不会跑这条命令——它只是输出了文字。你得手动复制到终端、执行、再把输出复制回对话框。这个"人工中转"就是最大的摩擦。

把它自动化掉，就是 agent loop。整个循环就靠两个信号驱动：第一，看 LLM 返回的 stop_reason 是不是 tool_use——如果是，说明它想调工具，那就执行工具、把结果塞回 messages，继续循环；如果不是，说明它要直接回答，那就返回文本结束。

看右边这段代码，不到 30 行，就是一个能跑的 agent 内核。注意模型决定"调什么、调不调"，代码只负责"执行、喂回去"。

顺便提一句——Claude Code 真实的 query.ts 有 1729 行，但核心还是这 30 行的循环，剩下的全是各种保护机制。这也是后面 19 课要讲的东西。

这一页我们做个对比——教学版的 30 行循环，和 Claude Code 真实的 1729 行，到底差在哪？

左边教学版做了几个简化：第一，它信任 stop_reason 字段；第二，整个 state 就是一个 messages 数组；第三，工具串行执行；第四，没有任何错误恢复和压缩。

右边 Claude Code 就不一样了。最关键的一点——它不信 stop_reason，而是用 needsFollowUp 主动去扫 response 里有没有 tool_use block。为什么？因为流式输出下 stop_reason 不可靠。然后它的 State 对象有 10 个字段，追踪轮次、恢复次数、压缩状态这些。工具能并发，有完整错误恢复。

底下的启示要记住——核心循环极其简单，复杂度全在保护机制上。这 19 课要做的，就是一层一层把这些保护机制加回去。但不管加多少层，那个 while True 的骨架，从头到尾没变过。

s02 TOOL USE 工具分发 · 扩展模型能触达的边界

加一个工具，只加一行

“循环不用动，新工具注册进 dispatch map 就行”

为什么不能只有 bash

只有 bash 时，"读文件"得翻译成 cat，"查找"得翻译成 find——浪费 token、翻译出错。专用工具去掉这层摩擦。

加一个工具 = 两个入口

  • TOOLS 数组加一项（声明给模型）
  • TOOL_HANDLERS 字典加一项（实现）
  • 循环一行没动

文件工具被 safe_path 保护；bash 仍未保护（留给 s03）。

# s02 · dispatch map 取代硬编码 TOOL_HANDLERS = { "bash": run_bash, "read_file": read_file, "write_file": write_file, "edit_file": edit_file, "glob": glob, } # 主循环里——只换一行： output = TOOL_HANDLERS[block.name](**block.input)

CC 进阶：工具 5 步校验管线 = Zod schema → validateInput → PreToolUse hooks → permission → call()。并发靠 isConcurrencySafe(input) 按输入判断，连续安全工具并行成批。

s02。格言——加一个工具，只加一行。

先问个问题：s01 只有 bash，不是也能干活吗，为什么还要别的工具？因为只有 bash 的时候，模型说"读一下那个文件"，它得自己翻译成 cat 命令；说"找一下所有 py 文件"，它得翻译成 find。这就既浪费 token，又容易翻译出错。给它专用的 read_file、glob 工具，就去掉了这层摩擦。

那怎么加？非常简单。看右边代码：你维护一个 TOOL_HANDLERS 字典，工具名映射到函数。加一个新工具，就是在 TOOLS 数组里声明一项、在 handlers 字典里加一项——主循环一行都不用改，循环里那行高亮的 dispatch 不变。

这里文件工具已经被 safe_path 保护了，但 bash 还是裸的——这个坑留给 s03。底下那张卡片提了 Claude Code 的进阶：它有个 5 步校验管线，并发是按输入判断的，连续安全的工具会并行成批执行。

s03 PERMISSION 权限 · 工具执行前的一道门

先划边界，再给自由

“工具执行前先做权限判断——安全靠代码，不靠模型信任”

痛点

s02 的 bash 没限制。模型说"清理一下项目"，它可能真的跑 rm -rf /。安全必须靠代码保证，不能靠模型自觉。

三道门，固定顺序

①
deny 名单

命中直接拦下，硬拒绝。

②
规则匹配

命中则升级到门 ③（上下文相关）。

③
用户审批

弹出来问：allow / deny。

都不命中 → 直接执行。日常 90% 操作直通。

# s03 · 循环里只多一行 for block in resp.content: if block.type == "tool_use": check_permission(block) # ← 新增 out = TOOL_HANDLERS[block.name](**block.input) results.append({"type":"tool_result", "tool_use_id": block.id, "content": out})

CC 进阶：4 种结果（allow/deny/ask/passthrough）；规则来自 8 个来源按优先级合并；子 agent 权限模式 'bubble'——弹窗冒泡到父终端。

s03，权限。格言——先划边界，再给自由。

为什么要权限？回到 s02，那个 bash 工具是完全没限制的。模型如果说"帮我清理一下项目"，它完全可能直接跑 rm -rf。这种事必须靠代码拦住，不能靠模型自觉——模型再聪明，也不能把安全寄托在它"这次不会犯错"上。

做法是三道门，按固定顺序走。第一道 deny 名单，命中直接拦；第二道规则匹配，命中就升级到第三道；第三道是用户审批，弹出来问你 allow 还是 deny。如果三道门都没命中，就放行——所以日常 90% 的读操作都是直通的，只有少数操作会停下来问你。

实现上特别优雅，看右边——主循环里只多了一行 check_permission(block)。

Claude Code 比这复杂：4 种结果而不是 3 种，规则来自 8 个来源按优先级合并。还有一个亮点——子 agent 的权限模式叫 bubble，意思是它需要审批时，弹窗会冒泡到父 agent 的终端去问。

痛点

每加一个检查（日志、权限、git add、通知）就往循环里塞 if——循环很快膨胀。正确的做法：循环保持稳定，扩展挂在循环外。

⏺

s04，hooks。格言——挂在循环上，不写进循环里。

痛点很好理解。你每加一个检查——打个日志、做个权限、跑个 git add、发个通知——就往主循环里塞一个 if。塞几次循环就膨胀得没法看了。正确的设计是：循环保持稳定，所有扩展挂在循环外面。

这里有个漂亮的重构——s03 那个 check_permission，现在整个搬成了一个 PreToolUse hook。主循环里那行权限检查可以删掉了，完全解耦。这就是 hook 的价值：能力在涨，循环不变。

s05 TODOWRITE 规划 · 动手前先想清楚

为什么会偏

  • 不增加执行能力——它不能读、不能跑，只是个内存里的状态清单

s05，TodoWrite。格言——没有计划的 agent，走着走着就偏了。

为什么 agent 会偏？长任务里，上下文被一堆工具结果填满，最开始那个目标就被稀释了。比如一个 10 步的重构，agent 做到第 3 步的时候，第 4 到 10 步可能已经被挤出注意力窗口了——它就忘了后面还要干嘛。

底下提了 Claude Code 的细节：它其实有两套并行的任务系统——V1 就是这个内存 TodoWrite，V2 是 s12 要讲的落盘 Task System，两套用一个开关切。这说明：规划是个被反复打磨的核心能力。

s06 SUBAGENT 子 Agent · 上下文隔离

大任务拆小，每个拿到的都是干净上下文

“子 agent 用独立 messages[]，不污染主对话”

痛点

修 bug 时为了追调用链读了 30 个文件——上下文被污染，agent 忘了原始问题。像"开个新终端"去调查，关掉，只把结论带回来。

task 工具 · 四个设计决策

  • 全新 messages[]——隔离，不继承历史
  • 只返回结论——extract_text(last_message)，不带回整个列表
  • 禁止递归——子 agent 没有 task 工具
  • 安全 hook 仍然生效——权限不因隔离而失效
# s06 · 派生子 agent def spawn_subagent(description): sub_messages = [{"role":"user", "content": description}] for _ in range(MAX_ROUNDS): # 30轮上限 resp = client.messages.create( system=SUB_SYSTEM, # 自己的prompt messages=sub_messages, tools=SUB_TOOLS) # 无 task 工具 sub_messages.append(resp) if resp.stop_reason != "tool_use": return extract_text( sub_messages[-1]) # 只返回文字结论 sub_messages.append(run_tools(resp))

CC 三种模式：① Normal 全新 messages；② Fork 共享 prompt cache（字节级一致前缀）；③ General-purpose。还有 run_in_background 异步路径（s13 讲）。

s06，子 agent。格言——大任务拆小，每个拿到的都是干净上下文。

痛点很常见。你在让 agent 修一个 bug，它为了追调用链去读了 30 个文件。读完之后，这 30 个文件的内容全塞在上下文里，把原始问题挤没了——agent 忘了自己一开始要干嘛。这就像你在终端里查东西，查着查着迷失了。

解法就是子 agent——像"开个新终端"一样，把调查任务派出去，子 agent 用干净的上下文去干，干完只把结论带回来，中间读的那些文件全丢弃。

四个设计决策记一下：第一，全新 messages 数组，不继承历史；第二，只返回最后一条的文字结论，不把整个 messages 列表带回来；第三，禁止递归——子 agent 没有 task 工具，不能再生子 agent；第四，虽然上下文隔离，但安全 hook 还是生效的，权限不会因为隔离而失效。

Claude Code 有三种模式，最值得一提的是 Fork 模式——它故意让父子 system prompt、tools、messages 前缀字节级一致，目的是命中 prompt cache，不是为了隔离。

s08 CONTEXT COMPACT 压缩 · 干净的记忆，无限的会话

上下文总会满，要有办法腾地方

snip_compact

L2

s08，上下文压缩。这是比较难的一课。格言——上下文总会满，要有办法腾地方。

agent 读了一堆文件之后，prompt 会超长，API 直接返回 prompt_too_long 拒绝调用。不做压缩，agent 在大项目里根本没法干活。

解法是一个四层流水线，便宜的先跑，贵的后跑。从轻到重：

L1 snip_compact——直接丢掉中间消息，只留头 3 条和尾 47 条。L2 micro_compact——把旧的 tool_result 替换成占位符。L3 tool_result_budget——把超过 200KB 的大输出落盘到磁盘，只在上下文里留个预览。L4 compact_history——前面三层都不够用时，才花一次 API 调用，让 LLM 做完整总结。

注意 L1 到 L3 是零 API 调用的，纯文本和结构操作；只有 L4 才花钱。还有一个 reactive_compact 是 413 错误时的紧急兜底。

底下这张卡片特别重要——顺序不能乱。budget 必须在 micro 之前跑，否则 micro 会先把大内容丢了，你想落盘都来不及了。还有一个熔断机制：连续 3 次压缩失败就停，避免死循环。

s09 MEMORY 记忆 · 跨压缩、跨会话

记住该记的，忘掉该忘的

为什么压缩不够

存储 + 索引

  • .memory/*.md 文件 + YAML frontmatter

②

③

s09，记忆。格言——记住该记的，忘掉该忘的。

为什么 s08 的压缩还不够？因为压缩会丢细节。比如"用户用 tabs 缩进"这种关键信息，压缩后可能变成"用户有风格偏好"——丢了具体内容。而且新开一个会话，连那个压缩总结都没了。

所以需要一层独立的记忆系统——它不参与压缩，跨会话存活。做法是文件仓库加索引：把记忆存成 .memory 目录下的 markdown 文件，每个带 YAML frontmatter；再维护一个 MEMORY.md 索引，常驻在 system prompt 里（这部分可缓存）。

关键设计是两层加载：第一层，索引永远在 system prompt，所以 agent 知道"我有哪些记忆"；第二层，真正用到时，用一次轻量 LLM 侧查询，选出不超过 5 个最相关的文件，把内容注入到当前用户消息里。这样既不会撑爆上下文，又能在需要时拿到细节。

还有提取和整理两个子系统。提取是每轮结束时让 LLM 判断"有没有值得记的新东西"，只在真新的时候才写。整理在 Claude Code 里叫"Dream"——像做梦一样，达到阈值就把碎片记忆合并成结构化的。

一个细节：Claude Code 用 Sonnet 模型自己来选记忆，不用 embedding——这个思路很巧。

s10 SYSTEM PROMPT 提示 · 运行时组装

prompt 是组装出来的，不是写死的

“分段 + 按需拼接 + 缓存”

写死 prompt 的三个问题

  • 每个项目都要重写一遍
  • 改一处可能全局冲突
  • 每次请求都带全部内容——token 浪费

解法：分段 + 按状态拼接

PROMPT_SECTIONS 字典：identity / tools / workspace / memory。assemble_system_prompt(context) 按真实状态加载（文件存不存在、工具注册没），不靠关键词猜。

为什么不全加载：system prompt 每轮都计费；信息越少越聚焦，无关指令就是噪声。

# s10 · 运行时组装 + 缓存 PROMPT_SECTIONS = { "identity": lambda ctx: IDENTITY, "tools": lambda ctx: build_tools_doc(), "workspace": lambda ctx: ( ws_doc if ctx.get("cwd") else ""), "memory": lambda ctx: ( mem_idx if ctx.get("has_memory") else ""), } def get_system_prompt(ctx): key = json.dumps(ctx, sort_keys=True) if key not in _cache: # 避免重拼 _cache[key] = "\n\n".join( s(ctx) for s in PROMPT_SECTIONS.values()) return _cache[key]

CC：~20-30 段，static/dynamic 用 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 切分，静态部分命中全局 cache block；mcp_instructions 是唯一易变段。

s10，系统提示词。格言——prompt 是组装出来的，不是写死的。

写死的 prompt 有三个问题：每个项目要重写、改一处可能全局冲突、最严重的是——每次 API 请求都把全部内容带上，纯粹浪费 token。

解法是把 prompt 拆成段，比如 identity、tools、workspace、memory，然后按真实状态拼接。注意是按真实状态，不是靠猜消息里的关键词。比如当前工作目录有内容，才加载 workspace 段；有记忆系统，才加载 memory 索引。

右边代码还展示了缓存技巧——用 json.dumps 把 context 当 key，避免重复拼接字符串。这个缓存跟 Claude Code 的 API 级 prompt cache 不是一回事，这里只是避免重复拼字符串。

底下提了 Claude Code 的做法：它有 20 到 30 个 prompt 段，用一条 boundary 把静态和动态分开，静态部分命中全局缓存块。唯一的易变段是 mcp_instructions——这正好接上 s19 要讲的 MCP。

核心理念：信息越少越聚焦，无关指令就是噪声。

s11 ERROR RECOVERY 韧性 · 遇错分类恢复

错误不是终点，是重试的起点

“升级 token、压缩上下文、切换模型——然后 continue 回循环”

① max_tokens

输出被截断。先升级 8K→64K（messages 不变）；仍截断则 append 输出 + 续写提示，最多续 3 次。

② prompt_too_long

上下文太长。reactive_compact 后重试；若已压缩过，放弃（不会再缩了）。

③ 429 / 529

限流/服务端错误。指数退避 + 抖动：min(500×2ⁿ, 32000) + rand(0~25%)，最多 10 次；连续 3 次 529 → 切 FALLBACK_MODEL。

# s11 · try/except 包住 LLM 调用 try: resp = client.messages.create(...) except MaxTokensError: if not state.escalated: state.max_tokens = 64000 # 升级 else: append_continuation() except PromptTooLong: reactive_compact(messages) except RateLimitError: backoff_sleep(attempt) continue # 回到 while True

RecoveryState 追踪

升级次数、恢复次数、续写次数——防止无限重试。

CC 的精细之处

  • 13+ 种原因/转换码
  • 收益递减检测：续写 3 次每次增量 <500 token → 停（没真输出）
  • 服务端 Retry-After 头优先

s11，错误恢复。格言——错误不是终点，是重试的起点。

生产环境里 API 报错是常态：输出被截断、上下文太长、限流、服务端 5xx。一个不处理错误的 agent，就像一辆一碰就熄火的车。

做法是把 LLM 调用包在 try/except 里，按错误类型路由到三条恢复路径之一，然后 continue 回到主循环。看上面三张卡片：

第一条 max_tokens——输出被截断。先升级 token 上限，从 8K 升到 64K，messages 不用动；如果还截断，就把已有输出 append 回去，加个"请继续"的提示，最多续写 3 次。

第二条 prompt_too_long——上下文太长。调 s08 的 reactive_compact 压缩后重试；如果已经压缩过了还报这个错，就放弃，因为不可能再缩了。

第三条 429 或 529——限流或服务端错误。指数退避加随机抖动，最多重试 10 次；如果连续 3 次都是 529，就切换到备用模型。

Claude Code 做得更细，有 13 种以上的错误转换码。我最喜欢的是它的收益递减检测——如果续写了 3 次，每次输出增量都不到 500 token，说明模型其实没在产出真东西，直接停。这种细节决定了 agent 到底好不好用。

s12，任务系统。这课很重要，是后面多 agent 协作的基础。格言——大目标拆成小任务，排好序，持久化。

为什么要它？因为任务有顺序——你不能先盖屋顶再打地基。任务之间形成依赖关系，本质是一个有向无环图，DAG。每个任务带一个 blockedBy 字段，列出它依赖哪些前置任务。

它跟 s05 的 TodoWrite 区别在哪？TodoWrite 是会话内的内存清单，关掉就没了；这个是落盘到 .tasks 目录的 JSON 文件，跨会话可恢复，而且带依赖顺序。

看状态机：pending 到 claim 变成 in_progress，到 complete 变成 completed。claim 的时候会检查 can_start——所有 blockedBy 依赖都必须是 completed 才能开始。complete 之后还会扫描下游，把因为这条任务完成而新解锁的任务放出来。

底下提了 Claude Code 的并发安全设计——它用 proper-lockfile 给每个任务文件加锁，还有个 highwatermark 文件防止 ID 被复用。这些细节在多 agent 抢任务时非常关键。

s13，后台任务。格言——慢操作丢后台，agent 继续思考。

这课的比喻特别好——洗衣机。你不会把衣服丢进洗衣机然后盯着它等 30 分钟吧？你会去干别的事。agent 也一样。npm install 要跑好几分钟，如果 agent 傻等着，LLM 那边还在按 token 计费，纯浪费。

怎么判断要不要丢后台？看耗时——文件读取是毫秒级，git status 大概 1 秒，这些同步等就行；分钟级的操作就该丢后台。

实现上是守护线程。模型在 bash 工具上加一个 run_in_background 参数，明确说"这个丢后台"。然后 start_background_task 起一个 daemon 线程跑命令，立即返回一个 bg_id 和占位结果。下一轮 collect_background_results 把完成的结果收集起来，包成 task_notification 注入进去。

这里有个容易踩的坑——通知不能复用原来那个 tool_use 的 id。为什么？因为协议里一个 tool_use 只能配一个同步的 tool_result，已经返回过占位符了。通知必须是独立注入的下一条消息。这种语义细节决定了你的 agent 会不会报奇怪的错误。

s14，cron 调度。格言——定时触发，不需要人推。

比喻是闹钟。闹钟设了 7 点，到点就响，不管你看没看着它。很多任务是周期性的——"每天 9 点跑测试"、"每 30 分钟查一下 CI"，这些不该每次都靠人去推一把。

这是个四层模型，生产者和消费者解耦。第一层，调度守护线程每秒轮询一次，检查哪个 cron 表达式匹配当前时间。第二层，匹配的任务进队列。第三层，队列处理器判断：队列非空、且 agent 空闲时，才启动一轮 agent loop。第四层，消费者往消息里注入一条 Scheduled 消息。

为什么要用队列和锁解耦？因为调度线程和 agent 主循环是两个不同的节奏——调度是每秒检查，agent 是按轮次跑。中间用 cron_queue 和 agent_lock 做缓冲，互不干扰。

Claude Code 在这上面加了很多生产级细节：抖动，用哈希决定每个任务提前一点点触发，避免所有任务都在 :00 或 :30 集中触发把服务打挂；7 天自动过期；最多 50 个任务。这些都是踩过坑才有的设计。

s16，团队协议。格言——队友之间要有约定。

s15 的协调是松散的——大家往邮箱塞消息就行。但有两个场景，松散协调会出问题。

第一个，关机。你不能直接 kill 一个队友线程，因为它可能正写到一半文件，直接杀掉就留下损坏的数据。需要握手——先发关机请求，等它处理完、回复确认，再真正停。

第二个，计划审批。队友准备做一个有风险的大重构，应该先把计划发给 Lead 审一遍，Lead 同意了再动手。

这两个场景本质是同一种结构——request 到 response，靠一个 request_id 串起来，带一个 pending/approved/rejected 的状态机。

右边代码看 ProtocolState。关键细节是 match_response 不仅要按 request_id 关联，还要校验 type——否则一个 shutdown 的 response 可能误批准了一个 plan_approval 请求。这种类型校验防的就是"串台"。

扩展性也很好——加一个新协议，就是在 dispatch_message 里加一个 if 分支。Claude Code 的关机是三段式：请求、批准或拒绝、最后还有个 teammate_terminated 通知，确保双方状态一致。

T1

s07，技能加载。格言——用到时再加载，别全塞 prompt 里。

痛点很直白。想象一下，你把 React 风格指南 2000 行、SQL 指南 1500 行、API 设计规范 3000 行，全塞进 system prompt——每次 API 调用都带 6500 行，但 99% 的时候跟当前任务根本无关，纯浪费。

解法是两层设计。第一层目录，启动时扫 skills 目录，解析每个技能的 YAML frontmatter，只取 name 和 description，注入大概 100 token 每个技能的目录进 system prompt。这部分量很小，而且可缓存——agent 至少知道"我有哪些技能可用"。第二层内容，真正用到时，调 load_skill，把那个 2000 token 的完整 SKILL.md 作为 tool_result 注入进来。这就是按需付费。

有个安全点——加载技能是通过注册表查表，不是用文件路径加载。这避免了路径穿越攻击，比如有人构造个恶意路径想去读 /etc/passwd。

这个模式跟 s09 的记忆、s10 的 prompt 组装，其实是同一个思想的不同应用：索引常驻，内容按需。

CC：6 种传输（stdio/sse/http/ws/sse-ide/sdk）；本地远程分批并发（3+20）；OAuth 2.0+PKCE；配置优先级 5 层。

s19，MCP 插件。格言——外接工具，标准协议。

为什么需要 MCP？前 18 课所有工具都是手写的。但你很快要接 Jira、接部署系统、接 Notion 知识库——你不可能为每个外部服务都重写一遍工具代码。MCP，也就是 Model Context Protocol，解决的就是这个：定义一个标准协议，服务端只要实现了它，agent 就能调用，不管服务端是 Python、Go 还是 Node 写的。

三步流程。第一步 connect_mcp，连接服务器并发 tools/list 去发现它提供哪些工具。第二步 assemble_tool_pool，把内置工具和 MCP 工具合并成一个池子——关键是命名空间，用 mcp__服务器名__工具名 的格式，避免不同服务器的同名工具冲突。第三步调用，发 tools/call，对 agent 来说，MCP 工具跟内置工具完全没区别。

底下有个重要的取舍——连接 MCP 之后工具池变了，s10 辛苦维护的 prompt cache 必须丢掉，否则缓存的工具列表是过时的。这是一个典型的"正确性优先于性能"的决策。

Claude Code 支持 6 种传输方式，本地和远程服务器分批并发连接，还有 OAuth 认证。MCP 现在是 agent 生态的事实标准。

s20，终点章。格言——机制很多，循环一个。这一页是整门课的高潮。

真实的 agent 不会一次只用一个机制。难点不在于堆功能，而在于看清楚每个机制挂在循环的哪个位置。这张图就是答案。

我们顺着用户输入往下走。先是 UserPromptUse hooks 做输入校验。然后是预 LLM 阶段——cron 队列、后台通知、压缩流水线，还有 memory、skill、MCP 的 prompt 组装，全在这里准备好。

然后错误恢复包住整个 LLM 调用——429、529 重试，max_tokens 升级，prompt too long 就压缩。真正调 LLM 拿到 response。如果没 tool_use 就 return，触发 Stop hooks。

如果有 tool_use，就进 PreToolUse hooks、权限检查、工具分发，后台任务也在这里分流。执行完触发 PostToolUse hooks。最后把 tool_result 或 task_notification append 回 messages，loop 回去。

看到了吗？s01 那个 30 行循环的骨架，一个字没改。所有机制都是挂在外面的钩子。这就是 harness 工程的核心洞察。
